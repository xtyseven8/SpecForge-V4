# 项目上下文协议

## 你的职责
执行任何 Skill 或任务前，必须：
1. 读取 `specs/` 目录下的所有文档
2. 扫描 `specs/prototype/` 目录下的所有 html 原型文档（如存在且非空）
3. 扫描代码结构，了解目录和已有代码
4. 确认理解项目做什么、用什么技术、有什么规范、UI 长什么样

## 为什么重要
你在新窗口中对项目一无所知。用户花了大量时间构建项目，选择技术栈，制定规范。不先了解就开始工作，会浪费他们的努力。

跳过上下文建立的真实代价：
- 用错技术栈 → 用户需要重写你的代码
- 违反项目规范 → 用户需要花时间修正你的错误
- 重复造轮子 → 用户会质疑你的专业性
- 破坏现有架构 → 用户可能需要回滚整个项目

## Red Flags

以下想法意味着你在合理化跳过上下文建立——停下来：

| 想法 | 现实 |
|------|------|
| "我先快速看一下代码" | 先读文档，文档告诉你代码为什么这样写 |
| "这个功能很简单，不需要了解项目" | 简单功能也会违反规范，先读 specs/ |
| "我记得这个项目" | 项目在演进，读当前版本 |
| "用户没说要读文档" | 这是你的职责，不是用户的请求 |
| "先写代码，有问题再改" | 返工成本远高于读文档成本 |
| "技术栈我猜得到" | 猜测不是理解，读 specs/技术栈选型.md |

## 检查协议
生成任何回复前，自问：

1. **Context Check**：我读过 `specs/` 目录了吗？
2. **Prototype Check**：`specs/prototype/` 目录存在且非空吗？如果是，我读过里面的 html 原型了吗？
3. **Stack Check**：我知道项目用什么技术栈吗？
4. **Convention Check**：我知道项目的命名、错误处理、API 风格约定吗？

任一为否 → 先读取相应文档，再开始工作。

## 原型文档协议（四份文档驱动）

SpecForge V4 在「需求/技术方案/任务规划」三份文档驱动的基础上，新增**原型文档**作为第四份驱动文档，确保前端开发的样式、排版、布局与原型一致。

### 路径与判定

- **原型目录**：`specs/prototype/`
- **判定规则**：
  - 目录存在且包含至少一个 `.html` 文件 → **四份文档驱动模式**，原型是 UI 实现的真相源
  - 目录不存在或为空 → **三份文档驱动模式**（回退），按原有流程执行

### 原型文档的作用

| 环节 | 原型文档的作用 |
|------|--------------|
| 需求澄清 | 参考原型确认交互细节、页面元素、用户操作路径 |
| 技术设计 | 对照原型设计组件结构、样式规范、布局方案、响应式断点 |
| 任务规划 | UI 相关任务的验证标准必须包含原型一致性检查点 |
| 编码实现 | 实现 UI 时必须对照原型，样式/布局/排版以原型为准 |
| 代码审查 | 增加「原型一致性」审查维度，验证实现与原型视觉一致 |
| 功能迭代 | UI 变更时同步更新原型文档，保持原型与实现一致 |

### 原型一致性原则

1. **原型是 UI 真相源**：当实现与原型冲突时，以原型为准（除非用户明确说明修改原型）
2. **样式精确还原**：颜色、字体、间距、圆角、阴影等视觉属性必须与原型一致
3. **布局结构对应**：DOM 结构和布局方式应与原型对应，不随意改变组件层级
4. **交互行为一致**：按钮点击、表单提交、动画过渡等交互行为与原型描述一致
5. **响应式一致**：响应式断点和布局变化与原型一致
